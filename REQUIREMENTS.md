# 要件定義（日本語版）

## 1. ビジョン

**すでに払っているフロンティアモデル**で論文を読む——2個目のサブスクを重ねず、per-tokenのAPI課金もせずに。左に本物のPDF、右にAIが組み立てた構造化要約、動かすのは自分のコーディングエージェント。

そしてこの製品の**本当の差別化点は「論文とコードの橋」**：VS Codeにしか無い「ユーザーのコードベース＋コードを理解するエージェント」を活かし、論文↔コードを、どのペーパーリーダーより密に繋ぐ（→ §4 北極星）。

## 2. 対象ユーザー

フロンティアAIサブスクを基盤インフラとして持つ（or これから持つ）学生・研究者——Claude Pro/Max、ChatGPT/Codex、あるいは**無料**のGoogleアカウント（Gemini CLI）。ただし以下を拒む層：

- アプリごとに**2個目の定額サブスク**を払う（例：alphaXiv Pro）
- サブスクの上に**per-tokenのAPI課金**を乗せる（BYOキー系ツール）

OSS・採用優先。お金のないユーザーも明確に対象（Gemini CLIの無料枠で$0運用が可能）。本命機能（論文↔コード）の最も濃い受益者は**ML/CS/システム系の研究者・院生**＝VS Codeに住んでいる層。

## 3. なぜ存在するか（楔）

既存の選択肢はどれも“3回目の支払い”を強いる：

| 選択肢 | ベースサブの上に乗る費用 |
|---|---|
| alphaXiv Pro | 2個目の定額サブスク |
| PapersGPT / Open Notebook（BYOK） | per-tokenのAPI課金 |
| 無料枠（alphaXiv無料・NotebookLM） | なし。ただし中位モデルに格下げ（Gemini Flash / Sonnet） |

**ベースサブスクに既に含まれるフロンティア枠を、論文読みに、追加コストゼロで使えるものが一つも無い。** その空白がこの製品。

さらに深い差別化が**論文↔コードの橋**：alphaXiv・NotebookLM・Zoteroには**コードベースが無い**ので構造的に追えない。バックエンド（CC/Codex/Gemini CLI）はそもそもコード理解に最適化されたエージェントなので、論文↔コードは**ホームグラウンド**。

## 4. スコープ

### v1（MVP — まず歩く骸骨、その後に薄く磨く）
- 生PDF表示（PDF.js）。図・レイアウトを保持。
- 同梱skillが生成する**全体の構造化要約**を右ペインに描画。
- バックエンド非依存：Claude Code / Codex / Gemini CLI を1つのAgent Skillで。
- OAuth不使用の**ファイル境界**。拡張は純粋なレンダラー。
- 完全ローカル。PDF＋要約ファイルがあればオフライン描画。

### 北極星（v2+）— 論文↔コードの橋
- **実装箇所の特定**：要約の式/アルゴリズムを指す → エージェントが自リポジトリの対応コードを探す。
- **整合チェック**：論文記述と自分の実装を突き合わせ、差分を指摘。
- **節から実装**：論文セクションを文脈に、エージェントがリポジトリにコードを書く（コーディングエージェントの本業）。
- **公式repoと並読**：repoをワークスペースにcloneし、論文と相互参照。
- 要約アンカーをPDFページだけでなく**コード位置**にも張る（スキーマに `code linkage` 次元を後付け）。
- *一部はv1でも“勝手に”効く*：CCは既にリポジトリ内にいるので、v1の要約を文脈に「これ実装して」「この関数は論文のどの式？」は追加実装ゼロで可能。拡張は後からそれを束ねる役を担う。

### その他の先送り（v2+）
- **ハイライト→質問/まとめ**（一節を選択→AI）。*生PDF上での確実なハイライト取得が難所のため先送り。*
- 任意の**Markdown描画**モード（ハイライトが容易・図の忠実度は微減）。
- 要約項目からの**ジャンプ**（ページアンカーは既にスキーマに有り）。
- 複数論文のライブラリ表示、論文横断Q&A。
- 任意のデスクトップアプリ接点。

## 5. 機能要件

- **FR-1** コマンド「BetaXiv: Open」が、アクティブ/選択したPDFに2ペインwebviewを開く。
- **FR-2** 左ペインがPDF.jsで本物のPDFを描画（全ページ・スクロール可）。
- **FR-3** 右ペインが現在のPDFの要約JSONを描画：Summary、主要貢献、目次（自動生成）、節ごとの散文（段落・箇条書き・数式・インライン図）、図（PDFページからの実画像クロップ）、用語集、未解決の問い、関連引用。
- **FR-4** 拡張は要約を `.betaxiv/summaries/<pdfのbasename>.summary.json` から探す。
- **FR-5** ファイル監視で、要約ファイル変更時に右ペインをライブ更新。
- **FR-6** 要約がまだ無ければ、右ペインに案内を表示（「このファイルに対してエージェントで `betaxiv-summarizer` skillを実行してください」）——拡張は**エージェントを起動しない**。
- **FR-7** 同梱skillは（ユーザーのエージェントが実行）、`papers/` のPDFを読み、スキーマ通り各論文を要約・検証し、出力ファイルを書き出す。

## 6. 非機能要件・制約

- **コンプライアンス** — 拡張はサブスクのOAuthやAPIキーを使わない。推論はすべてユーザー自身のログインで動く公式エージェント経由。（Anthropic 2026年2月/4月のToS、GoogleのGemini CLI ToSに整合。OpenAI/Codexは現状最も寛容。）
- **ライセンス** — 自前コードはMIT or Apache-2.0。抽出依存は `pdfplumber`/`docling`（MIT）、描画は `pdf.js`（Apache）。**PyMuPDF / MinerU / Marker は不可。**
- **配布** — OSS。VS Code Marketplace（無料）＋ Open VSX。収益化は**寄付のみ**（GitHub Sponsors / Open Collective ＋ 研究室/機関スポンサー枠）。有料バックエンド・ライセンスキー・アカウントなし。
- **プライバシー** — すべてローカル。既定でテレメトリなし。PDFも要約もマシンから出ない（推論はユーザー自身のエージェント）。
- **パフォーマンス** — 大きなPDFに対応。skillはページ範囲読み込みでエージェントのコンテキスト/トークン消費を抑える。
- **ポータビリティ** — 1つの `SKILL.md` が Agent Skills標準と `.agents/skills/` エイリアスで CC / Codex / Gemini CLI 横断で動く。拡張のインストールロジックが検出したエージェントごとに正しいディレクトリへ配置。

## 7. データ契約

skillと拡張の間のインターフェースは、単一の**バージョン付きJSON要約**。両側が `schema/summary.schema.v2.json` で検証する。形（v2.0）：

- `schemaVersion` — `"2.0"`
- `paper` — `sourcePath`, `title`, `authors[]`, `year`, `venue`, `date`
- `summary` — `tldr`, `keyContributions[]`, `sections[]`（`heading`, `page`, `blocks[]`）, `figures[]`（`label`, `caption`, `page`, `bbox`）, `glossary[]`, `openQuestions[]`, `relevantCitations[]`
- `generatedBy` — `agent`, `model`, `timestamp`

`blocks[]` は順序付き散文：`paragraph` / `bullets` / `formula` / `figure`（`label` で図カタログを参照しインライン配置）。`figure.bbox` は正規化 `[x0,y0,x1,y1]`（0..1, 左上原点）で、モデルは図の**位置のみ**を指定し、拡張が PDF ページから実画像をクロップする（図は描かない）。

`page`/`date` 等は必須だがnull許容。将来のジャンプ/ハイライトをスキーマ変更なしで可能にするため、今から持たせておく。

**v2.0 ハードカット（後方互換なし）**：`sections[].points[]` を `blocks[]` に置換、`figures[].bbox`・`paper.date`・`summary.relevantCitations[]` を追加。v1 要約は検証で弾かれ、skill 再実行で再生成する。

## 8. 対象外（v1）

- ハイライト・注釈・PDF内インタラクション。
- 拡張発のモデル/ネットワーク呼び出し。
- クラウド同期・アカウント・課金・テレメトリ。
- VS Code以外の接点。（ブラウザ拡張はサンドボックスでローカルのエージェント＋ファイル境界を載せられない。Zoteroのプラグイン環境も同様に限定的。サブスク再利用の楔はローカル/デスクトップ接点を必要とする。）

## 9. 成功基準

- **採用** — インストール数と週次リテンション。
- **不可欠性** — 消えたら困ると感じられること。
- **バックエンド網羅** — CC・Codex・Gemini CLI で動作確認済み。
- 寄付は*ボーナス*であってKPIではない。

## 10. 未解決の問い・リスク

- **プラットフォーム政策（最大リスク）。** Anthropic・Googleは第三者によるサブスクOAuth利用を禁止。我々はファイル境界＋人間トリガー設計で回避する。もし提供側が「製品のためにエージェントを駆動する」を第三者利用と再解釈したら見直す。Codexが現状最も寛容なバックエンド。
- **枠の食い合い** — フロンティアモデルで読むと、コーディングに使う同じサブスク枠を消費する。
- **ネイティブPDF読み vs 抽出** — Claude Codeのネイティブd PDF読みは文書化されているがプラットフォーム依存で不安定。バックエンドごとに、skillがネイティブ読みに頼るか pdfplumber/docling を一枚噛ますかを決める。
- **図の忠実度** — v2 で実装済み（モデルは図の位置=bbox のみ指定し、拡張が PDF ページから実画像をクロップ）。忠実度はモデルが出す bbox 精度に依存し、null/失敗時は caption へ graceful degrade。
